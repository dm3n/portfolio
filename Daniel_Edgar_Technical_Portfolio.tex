% ================= RESEARCH =================
\section{Original Research Contribution}
\subsection{Uncertainty Propagation in Tree-Structured Language Model Reasoning \quad\small\normalfont(\repo{uncertainty-propagation})}
Single-author paper formalizing how per-step errors compound across multi-step LLM reasoning.
Introduces the \textbf{Reasoning Tree Model (RTM)}, which unifies chain-of-thought, self-consistency,
and tree-of-thought as special cases, and derives the governing results from first principles:

\begin{itemize}
  \item \textbf{Chain success} $=\prod_i (1-\theta_i)$, decaying exponentially in depth (Theorem~3.1).
  \item \textbf{Tree majority vote} decays only \emph{sub}-exponentially in compute, $\exp(-\Omega(N^{1-1/(d+1)}))$ --- a qualitatively different reliability regime (Theorem~4.2).
  \item Closed-form \textbf{compute-optimal branching factor and depth} (Theorem~4.3); MLE for step-failure rates with delta-method confidence intervals on end-to-end success (Theorems~5.1--5.2).
  \item \textbf{Empirical validation} against GPT-5, Claude Opus 4.6, Claude Sonnet 4.6, and Gemini 3.1 Pro on GSM8K, MATH, and MMLU-Pro: predictions track observed accuracy to within \textbf{$\sim$1\% mean absolute error}. Signature result: at a fixed 48-step compute budget, a chain succeeds $\approx 0.2\%$ of the time while a depth-3 tree succeeds \textbf{97.9\%}.
\end{itemize}
This research is the design theory behind the production systems below: decompose work into short,
independently verified, tree-structured units rather than one long, exponentially decaying chain.

\section{Flagship Systems}

\subsection{Symphony --- autonomous software-delivery runner \quad\small\normalfont(\repo{symphony})}
A Jira-native, evidence-gated, supervised-autonomy runner that drives a product ticket from backlog
to a reviewed, evidence-backed pull request. \textbf{Built as CTO of Finsider and deployed live in
its engineering environment.} $\sim$2{,}300-line standard-library Python orchestrator.
\begin{itemize}
  \item \textbf{Jira as the state machine} --- ticket status and labels are the control flow; no external queue or database; crash-safe because state is re-derived from Jira each poll.
  \item \textbf{Per-ticket isolated Git workspaces}; acceptance-contract prompting of a coding agent; \textbf{Playwright} screenshot/video evidence capture.
  \item \textbf{Intent-classified human review} (question vs.\ rework vs.\ approval); \textbf{cross-repo scope critic}; \textbf{evidence-integrity gating}; \texttt{--force-with-lease} safe Git; recursive process-tree cleanup; bounded failure auto-retry.
  \item Dual deployment: macOS \texttt{launchd} (local) and an always-on GCP Compute Engine VM under \texttt{systemd}.
\end{itemize}
